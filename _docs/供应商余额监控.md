- 桌面 1440x900 与移动 390x844 的全部主视图已完成浏览器渲染、交互、控制台和横向溢出检查。

### 验证边界

- 开源项目契约通过固定响应 Fixture 和对应源码端点验证，不等同于任意第三方部署版本都兼容。
- DeepSeek、OpenRouter、VoAPI 和公网中转站的真实站点 Canary 仍需要用户提供只读测试凭据；没有凭据时不得声称完成线上实测。
- Docker Compose 配置已通过解析校验；当前开发机 Docker Desktop daemon 不可用，因此本次无法完成镜像构建和多容器 E2E。
- 外部 Secret Manager、Cherry Studio 等客户端的主动写入同步，以及 Octopus、AxonHub、Claude Code Hub 等实验平台仍属于文档标注的后续扩展，不计入当前 verified 适配器。

## 结论摘要

本模块不应只实现“保存多个 Key，然后逐个查询余额”。正确的产品边界是：

1. 统一管理上游连接和认证会话。
2. 从远端同步账户、API Key、可用分组和 Key 所属分组。
3. 明确区分账户余额、Key 额度、团队预算和累计用量。
4. 持续记录快照、计算消耗速度、预测耗尽时间并发送告警。
5. 在严格的安全控制下，将低余额、Key 失效和上游异常联动到 Sub2API 渠道。

原方案中影响主功能的问题已经在本文中修正：

- New API 增加用户 ID 请求头，不再只发送系统令牌。
- One API 家族直接使用 quota 作为剩余额度，不再默认减去 used_quota。
- 额度除数优先从站点的 /api/status 自动发现。
- 本地认证凭据与远端 API Key 分开建模。
- 账户余额和 Key 额度分开存储及汇总，避免重复统计。
- OpenRouter 改用当前官方的 /api/v1/key、/api/v1/credits 和 /api/v1/keys 契约。
- 批量任务改为固定 Worker 队列，单项失败不会中断整批任务。
- 配置和历史数据改用 SQLite 事务，不再依赖 config.json 与简单内存锁。
- 增加独立鉴权、AES-256-GCM、SSRF 防护、审计日志和强制 Secret。
- 状态模型支持负余额、无限额度、多币种、陈旧数据和部分能力缺失。
- VoAPI、Veloera 等分支使用独立适配器和能力探测，不再假设与 New API 完全兼容。

## 一、需求背景

Sub2API 会集成多个上游供应商或自建中转站，例如：

- Sub2API
- New API
- One API
- OneHub
- DoneHub
- DeepSeek
- OpenRouter
- LiteLLM
- 其他提供 OpenAI、Anthropic 或 Gemini 兼容接口的中转站

这些平台中可能同时存在以下资产：

- 账户充值余额
- 账户冻结余额
- API Key 独立额度
- API Key 所属路由分组
- 用户可用分组
- 团队或项目预算
- 套餐额度和重置周期
- 赠送余额与充值余额
- 不同币种余额

当账户数量增加后，管理员面临以下问题：

- 需要逐站登录查看余额和 Key 状态。
- 无法统一判断哪个账户即将耗尽。
- 多个 Key 属于同一账户时容易重复计算资产。
- 无法发现远端 Key 被禁用、过期或被改组。
- 无法把上游余额变化与 Sub2API 的渠道使用量对应起来。
- 余额不足时无法及时停用渠道或切换备用供应商。

因此需要一个常驻运行的管理员工具，统一完成资产发现、监控、分析、告警和受控自动化。

## 二、目标与非目标

### 2.1 核心目标

1. 统一管理不同平台的认证凭据和认证会话。
2. 自动发现平台类型、版本、额度单位和支持能力。
3. 同步远端账户、API Key、分组及 Key 额度。
4. 查询账户余额并按币种正确展示。
5. 保存历史快照，计算消耗趋势和预计耗尽时间。
6. 对余额、Key、分组、可用性和数据陈旧状态进行告警。
7. 输出 Prometheus 指标，并支持 Webhook、Telegram、邮件、Gotify 和 Bark。
8. 将供应商资产映射到 Sub2API 渠道，支持安全的停用、恢复和故障转移。

### 2.2 非目标

1. 不把所有 OpenAI 兼容接口都误认为具有余额查询能力。
2. 不通过模型调用接口推测账户充值余额。
3. 不默认执行充值、创建 Key、删除 Key 或修改分组等高风险写操作。
4. 不在前端返回或长期展示完整明文密钥。
5. 不把不同币种未经汇率换算直接相加。
6. 不承诺兼容无法获得源码、公开接口文档或真实实例验证的平台版本。

## 三、核心术语与数据语义

### 3.1 Provider Connection

供应商连接，表示一个可认证的远端站点实例。

示例：

- https://newapi.example.com 上的一个用户账户
- DeepSeek 官方账户
- OpenRouter Workspace 的 Management Key
- 自建 LiteLLM Proxy 的 Master Key

连接只保存认证信息和站点配置，不等同于远端 API Key。

### 3.2 Remote Account

远端账户，是余额的主要归属对象。

同一账户下可能有多个 API Key，但账户充值余额只统计一次。

### 3.3 Remote Key

从远端同步得到的 API Key 元数据。默认只保存：

- 远端 ID 或 Hash
- 名称和标签
- 脱敏 Key
- 状态
- 所属分组
- 额度、用量和到期时间

只有平台明确支持安全读取完整 Key，且管理员主动执行导出操作时，才临时获取明文。

### 3.4 Group 类型

分组必须区分以下语义：

| 类型 | 枚举 | 说明 |
|---|---|---|
| 账户所属组 | user_group | 用户在站点中的等级或计费组 |
| Key 路由组 | key_route_group | 某个 API Key 请求时使用的路由组 |
| Key 备用组 | key_backup_group | 主分组不可用时的备用路由组 |
| 上游账号组 | upstream_account_group | 中转站管理员维护的上游账号资源组 |
| 团队或项目 | team | LiteLLM 等网关的团队、项目或预算归属 |
| Workspace | workspace | OpenRouter 等平台的工作空间 |
| 本地标签 | local_tag | Provider Monitor 自己维护的标签，不属于远端 |

普通用户凭据通常只能读取 user_group、key_route_group 和可用分组。
upstream_account_group 通常需要远端管理员权限。

### 3.5 Balance Scope

所有余额或额度必须带作用域：

| 作用域 | 枚举 | 示例 |
|---|---|---|
| 账户余额 | account | New API quota、Sub2API balance、DeepSeek total_balance |
| Key 额度 | key | One API Token remain_quota、OpenRouter limit_remaining |
| 团队预算 | team | LiteLLM max_budget - spend |
| 套餐额度 | subscription | 月度套餐剩余额度 |

只有 account 作用域的金额可以进入账户资产汇总。
key、team 和 subscription 作用域单独展示，禁止与账户余额重复相加。

### 3.6 Capability

每个适配器通过能力标记说明实际支持范围：

~~~javascript
{
  accountBalance: true,
  multiCurrencyBalance: false,
  listKeys: true,
  revealKey: false,
  keyQuota: true,
  listGroups: true,
  keyGroup: true,
  backupGroup: false,
  usageHistory: true,
  priceCatalog: false,
  checkIn: false,
  credentialRefresh: true,
  writeOperations: false
}
~~~

前端只展示适配器实际支持的功能，不为缺失能力伪造空数据。

## 六、数据模型

### 6.1 provider_connections

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID | 本地连接 ID |
| name | TEXT | 自定义名称 |
| adapter_type | TEXT | 适配器类型 |
| base_url | TEXT | 标准化后的站点 URL |
| auth_mode | TEXT | system_token、jwt_refresh、api_key、management_key 等 |
| credential_id | UUID | 加密凭据引用 |
| remote_user_id | TEXT | New API、Veloera、VoAPI 等所需用户 ID |
| enabled | BOOLEAN | 是否启用监控 |
| capabilities_json | JSON | 最近一次能力探测结果 |
| fingerprint_json | JSON | 平台指纹、版本和额度单位 |
| tags_json | JSON | 本地标签 |
| note | TEXT | 备注 |
| last_sync_at | DATETIME | 最近同步时间 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 修改时间 |

同一个 base_url 可以有多个不同账户连接。

### 6.2 encrypted_credentials

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID | 凭据 ID |
| version | INTEGER | 加密格式版本 |
| algorithm | TEXT | aes-256-gcm |
| salt | BLOB | KDF Salt |
| iv | BLOB | GCM IV |
| ciphertext | BLOB | 密文 |
| auth_tag | BLOB | GCM Tag |
| created_at | DATETIME | 创建时间 |
| rotated_at | DATETIME | 最近轮换时间 |

明文可以包含：

~~~json
{
  "accessToken": "...",
  "refreshToken": "...",
  "apiKey": "...",
  "email": "...",
  "password": "...",
  "customHeaders": {
    "X-API-Key": "..."
  }
}
~~~

所有认证请求头都属于敏感凭据，禁止存放在未加密的 type_config 中。

### 6.3 remote_accounts

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID | 本地记录 ID |
| connection_id | UUID | 所属连接 |
| remote_id | TEXT | 远端账户 ID |
| display_name | TEXT | 账户名称 |
| user_group | TEXT | 账户所属组 |
| status | TEXT | active、disabled、unknown |
| metadata_json | JSON | 平台特定非敏感信息 |
| first_seen_at | DATETIME | 首次发现 |
| last_seen_at | DATETIME | 最近同步 |

connection_id + remote_id 必须唯一。

### 6.4 remote_groups

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID | 本地记录 ID |
| connection_id | UUID | 所属连接 |
| remote_id | TEXT | 远端分组 ID 或 Symbol |
| group_type | TEXT | user_group、key_route_group、team 等 |
| name | TEXT | 分组名称 |
| ratio | REAL | 倍率 |
| status | TEXT | active、disabled、removed |
| metadata_json | JSON | 平台扩展数据 |
| last_seen_at | DATETIME | 最近同步 |

### 6.5 remote_keys

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID | 本地记录 ID |
| connection_id | UUID | 所属连接 |
| remote_account_id | UUID | 所属远端账户 |

| name | TEXT | 名称 |

### 7.3 响应校验

适配器不能只检查 HTTP 401、403 和 5xx，还必须：

1. 将所有非 2xx 状态视为失败。
2. 单独识别 400、404、409、422 和 429。
3. 解析 HTTP 200 中的 success=false、code 非零等业务错误。

5. 保存脱敏后的响应摘要，禁止保存完整密钥。
6. 对分页接口设置最大页数和最大对象数。
7. 对 Content-Type、响应体大小和 JSON 深度设置限制。

## 八、平台适配器实现

### 8.1 Sub2API

#### 认证

推荐认证流程：

1. 首次由管理员完成邮箱密码登录。
2. 保存 access_token、refresh_token 和 expires_in。
3. 后续优先使用 refresh token 刷新。
4. Access Token 提前 60 秒刷新。
5. 收到 401 后只允许执行一次刷新并重试，防止循环。

已知限制：

- 开启 Turnstile 时，纯服务端首次登录可能失败。
- 开启 TOTP 2FA 时，需要管理员完成交互式登录。
- Backend Mode 下普通用户不可登录，但管理员账户可以使用。
- 如果无法自动登录，允许管理员粘贴已有 Access Token 和 Refresh Token。

#### 接口

|---|---|---|
| 登录 | POST /api/v1/auth/login | 返回 Token Pair |
| 刷新 | POST /api/v1/auth/refresh | 使用 refresh_token 换取新的 Token Pair |
| 账户余额 | GET /api/v1/user/profile | data.balance，USD |
| 冻结余额 | GET /api/v1/user/profile | data.frozen_balance |
| Key 列表 | GET /api/v1/keys | 分页返回 |
| 可用分组 | GET /api/v1/groups/available | 普通用户可访问 |

#### 凭据

- 用户 ID

#### 接口与语义

| 功能 | 接口 | 说明 |
|---|---|---|
| 平台状态 | GET /api/status | version、quota_per_unit、币种展示配置 |

- 可以通过 /api/usage/token 查询当前 Key 额度。
- 不能查询账户余额。
- 不能列出账户全部 Key。
- 不能获取该 Key 的远端 group 字段。

## 九、平台识别与能力探测

### 9.1 显式类型优先

管理员选择的平台类型优先于自动识别。自动识别只能作为建议，不能静默改变已有连接的适配器。

1. 标准化 Base URL。
2. 请求公开状态接口，例如 /api/status。

- adapter_type

- version
- response_schema_hash
- quota_per_unit
- currency display config
- detected_at

- 标记 contract_changed。
- 暂停高风险自动化动作。
- 继续保留最后一次成功数据。
- 触发兼容性告警。

## 十、余额、状态和汇总规则

### 10.1 状态定义

| 状态 | 条件 |
|---|---|
| pending | 从未成功查询 |
| normal | 可用余额高于警告阈值 |
| warning | 可用余额大于 0 且小于等于警告阈值 |
| critical | 可用余额小于等于 0 |
| unlimited | 平台明确返回无限额度 |
| stale | 最近成功数据超过允许陈旧时间 |
| error | 当前查询失败且不存在可用的历史成功值 |
| unsupported | 平台不支持该能力 |
| auth_expired | 凭据过期且无法刷新 |
| disabled | 连接或远端对象已禁用 |

查询失败但存在历史成功值时：

- 保留上次余额。
- 状态显示 stale 或 warning_stale。
- 单独展示本次错误。
- 汇总中可以按配置决定是否计入，并清晰标注数据时间。

### 10.2 阈值

支持以下规则：

- 全局默认阈值
- 每币种阈值
- 每连接阈值
- 每账户阈值
- 每 Key 阈值
- 固定金额阈值
- 余额百分比阈值
- 预计可用天数阈值
- Key 到期前天数阈值
- 数据陈旧时间阈值

### 10.3 汇总

默认看板分别展示：

- USD 账户余额小计
- CNY 账户余额小计
- 其他币种分项
- 可选的折算 USD 总额
- Key 剩余额度

- 套餐剩余额度

- 汇率来源

## 十一、任务调度、并发与重试

### 11.1 任务类型

- capability_probe
- account_balance_refresh
- remote_inventory_sync
- key_health_check
- usage_sync
- price_sync
- check_in
- alert_evaluation
- forecast_rebuild
- notification_delivery
- automation_evaluation

### 11.2 Worker 队列

node-cron 只负责投递任务，不直接执行网络请求。

固定 Worker 队列保证：

- 单个任务失败不会中断其他任务。
- 每个连接同类型任务互斥。
- 不同连接可以并发。
- 手动刷新可以复用正在执行的任务。
- 服务重启后可从数据库恢复未完成任务。

参考逻辑：

~~~javascript
async function runWorkers(items, concurrency, worker) {
  const queue = [...items]
  const results = []

  async function run() {
    while (queue.length > 0) {
      const item = queue.shift()
      try {
        results.push({ item, status: "fulfilled", value: await worker(item) })
      } catch (error) {
        results.push({ item, status: "rejected", error })
      }
    }
  }

  await Promise.all(
    Array.from({ length: Math.min(concurrency, queue.length) }, run)
  )
  return results
}
~~~

### 11.3 默认并发

- 全局网络并发：5
- 单站点并发：2
- 登录和刷新凭据并发：1
- Key 可用性探测并发：3

所有值可以配置，但必须设置安全上限。

### 11.4 重试

| 错误 | 策略 |
|---|---|
| 网络瞬断 | 最多 2 次指数退避 |
| HTTP 429 | 尊重 Retry-After，并加随机抖动 |
| HTTP 500、502、503、504 | 最多 2 次指数退避 |
| HTTP 401 | 仅允许刷新凭据后重试 1 次 |
| HTTP 403 | 不自动重试 |
| HTTP 400、404、422 | 不自动重试 |
| Schema 变化 | 不重试，触发 contract_changed |

定时任务增加 0 到 20% 随机抖动，避免所有站点同一时刻刷新。

### 11.5 熔断

连续失败达到阈值后进入熔断：

- 默认连续失败 5 次。
- 熔断 15 分钟。
- 熔断期间只允许管理员手动探测。
- 认证失败和 Contract 变化使用独立熔断原因。

## 十二、持久化与一致性

### 12.1 SQLite

使用 SQLite WAL 模式，替代 config.json。

原因：

- 支持事务和唯一约束。
- 支持并发读取。
- 避免定时刷新覆盖管理员刚保存的配置。
- 支持历史快照、告警、审计和任务恢复。
- 支持后续数据迁移。

### 12.2 写入策略

一次刷新使用事务：

1. 创建 check_run。
2. Upsert 远端账户、分组和 Key。
3. 写入余额与用量快照。
4. 标记本轮未出现的远端对象。
5. 更新连接的 last_sync_at。
6. 提交事务。
7. 事务提交后评估告警。

任何步骤失败则回滚本轮业务数据，但保留失败的 check_run。

### 12.3 数据保留

默认策略：

- 原始分钟级快照：30 天
- 小时聚合：180 天
- 日聚合：永久或按配置
- 任务明细：90 天
- 审计日志：365 天
- 通知投递：180 天

清理任务必须分批执行，避免长事务。

### 12.4 备份

- 支持 SQLite 在线备份。
- 导出文件默认不包含凭据。
- 包含凭据的灾备包必须二次加密并要求管理员重新输入导出密码。
- 支持 WebDAV、S3 兼容存储和本地目录。

## 十三、鉴权与安全

### 13.1 Provider Monitor 自身鉴权

默认复用 Sub2API 管理员身份：

1. 前端提交 Sub2API 登录请求或使用已有管理员 JWT。
2. Provider Monitor 调用 Sub2API 用户信息接口验证 Token。
3. 只允许 role=admin 或 root。
4. 后端建立 HttpOnly、SameSite=Strict 的短期会话。
5. 所有写操作要求 CSRF Token 和 Origin 校验。

可选认证模式：

- sub2api：推荐
- reverse_proxy：由受信反向代理注入管理员身份
- local_admin：仅在无法连接 Sub2API 时使用，密码使用 Argon2id

禁止 auth_mode=none。

### 13.2 网络暴露

Docker 默认仅监听本机：

~~~yaml
ports:
  - "127.0.0.1:${PROVIDER_MONITOR_PORT:-9871}:3200"
~~~

需要跨主机访问时，应通过 HTTPS 反向代理、VPN 或零信任访问层暴露。

### 13.3 凭据加密

- 算法：AES-256-GCM
- KDF：scrypt
- 每条凭据独立 Salt 和 IV
- 加密格式带 version
- Secret 必填，长度至少 32 个随机字符
- 禁止任何默认 Secret
- 支持密钥轮换和逐条重加密

环境变量：

~~~bash
PROVIDER_MONITOR_SECRET=必须由密码管理器生成
~~~

服务在 Secret 缺失时必须拒绝启动。

### 13.4 前端密钥处理

- 列表接口只返回 masked_key。
- 不提供普通的“点击复制完整 Key”。
- 明文导出要求重新验证管理员身份。
- 明文只在内存中存在，响应设置 Cache-Control: no-store。
- Reveal 操作写入审计日志。
- 页面失焦或 30 秒后清除临时明文。
- 禁止在日志、错误、埋点和 raw_json 中记录密钥。

### 13.5 SSRF 防护

所有远端请求必须经过统一网络策略：

- 仅允许 http 和 https。
- 已知云元数据地址始终禁止访问。
- 私网主机限制为空时，放行 localhost、环回、链路本地、组播及其他私网地址。
- 私网主机限制非空时，私网供应商必须匹配“设置与备份 -> 系统参数”中的主机名单。
- DNS 解析后检查所有 A 和 AAAA 地址。
- 每次重定向重新检查目标。
- 禁止 URL 中携带用户名和密码。
- 限制端口范围。
- 限制请求和响应大小。
- 禁止自定义 Host、Connection、Content-Length 等危险 Header。
- 记录目标 Host，不记录敏感 Query。

无论 Allowlist 如何配置，都永久禁止常见云元数据地址。

### 13.6 审计

记录：

- 登录成功和失败
- 新增、修改、删除供应商
- 凭据更新和轮换
- Reveal、导出和备份
- 手动刷新
- 告警确认
- 自动化启用、执行和回滚
- SSRF 拒绝

## 十五、趋势、分析与预测

### 15.1 余额变化趋势

- 近 24 小时
- 近 7 天
- 近 30 天
- 自定义时间范围
- 按连接、账户、币种和 Key 筛选

### 15.2 消耗速度

- 最近 1 小时平均消耗
- 最近 24 小时平均消耗
- 最近 7 天日均消耗
- 工作日与周末差异
- 峰值小时消耗

### 15.3 预计耗尽时间

~~~text
runway_days = available_balance / recent_daily_burn_rate
~~~

只有满足以下条件时才显示预测：

- 至少存在 3 个有效快照。
- 时间跨度达到配置阈值。
- 消耗速度为正。
- 数据未处于 stale。

同时提供：

- 线性预测
- 7 日移动平均
- 异常点排除后的稳健预测
- 预测可信度

### 15.4 异常检测

检测：

- 余额异常骤降
- 用量异常突增
- 长时间无消费但余额减少
- Key 用量与账户用量不一致
- 站点返回余额回退或重置
- 汇率或额度单位发生变化
- 远端 Schema 变化

## 十六、Key 健康与配置漂移

### 16.1 Key 可用性检测

分层检测：

1. 元数据检测：状态、到期时间和剩余额度。
2. GET /v1/models：验证认证和模型列表。
3. 最小文本调用：可选，使用低成本模型。
4. Tool Calling、Structured Output、Web Search 等能力探测：按需执行。

默认不执行产生费用的请求，付费探测必须显式开启并设置每日预算。

### 16.2 Key 漂移检测

对比两次同步结果：

- 新增 Key
- 删除 Key
- Key 被禁用
- Key 到期时间变化
- Key 主分组变化
- 备用组变化
- 额度从无限改为有限
- 模型权限变化
- IP 白名单变化

### 16.3 分组漂移

检测：

- 分组被删除或停用
- 倍率变化
- Key 指向不存在的分组
- Key 从低成本组切到高成本组
- 备用组与主组相同
- Sub2API 渠道映射的目标分组发生变化

## 二十一、受控自动化

### 21.1 自动化动作

可选动作：

- 余额不足时暂停对应 Sub2API 渠道
- Key 失效时禁用映射渠道
- 数据恢复后重新启用渠道
- 切换到备用供应商
- 调整渠道优先级或权重
- 触发外部充值工单 Webhook
- 提醒轮换凭据
- 创建或更新本地路由建议

### 21.2 安全原则

所有自动化默认关闭，并遵循：

- Read Only 默认模式
- Dry Run
- 明确的规则作用范围
- 最少连续命中次数
- 冷却时间
- 每日最大动作次数
- 变更前快照
- 回滚动作
- 审计日志
- Contract 变化时自动暂停

删除 Key、创建 Key、修改远端额度和充值不纳入默认自动化。

### 21.3 示例规则

~~~yaml
name: low-balance-disable-channel
when:
  balance_scope: account
  currency: USD
  available_lte: 3
  consecutive_matches: 2
then:
  action: disable_sub2api_channel
  cooldown_minutes: 60
  dry_run: true
recover:
  available_gte: 10
  action: enable_sub2api_channel
~~~

- SSRF 地址判断

### 28.2 契约 Fixture

每个平台保存脱敏 Fixture：

- 成功响应
- 认证失败
- 权限不足
- HTTP 200 业务失败
- 分页
- 空列表
- Schema 变化
- 无限额度
- 多币种

Fixture 必须标记：

- 来源项目
- Commit SHA 或版本
- 获取日期
- 对应接口

### 28.3 Mock Server

模拟：

- 延迟
- 连接断开
- 429 + Retry-After
- 5xx
- 401 后刷新成功
- 401 后刷新失败
- 部分分页失败
- 重定向到内网地址
- 超大响应体

### 28.4 Docker E2E 矩阵

对可部署的开源项目使用固定镜像或 Commit：

| 平台 | 验收内容 |
|---|---|
| Sub2API | 登录、刷新、账户余额、Key、分组、用户有效倍率、可选渠道模型价格 |
| New API | 系统 Token + User ID、余额、Key、分组 |
| One API | 余额、账户组、Token 额度 |
| OneHub | 余额、Token 主组和备用组 |
| DoneHub | 余额、Token 主组和备用组 |
| LiteLLM | Key、Team、Budget 和 Spend |

E2E 测试创建独立测试账户和最小测试数据。

### 28.5 真实站点 Canary

DeepSeek、OpenRouter 和 VoAPI 等需要真实凭据的平台通过可选 Canary 测试：

- 凭据仅存储在 CI Secret。
- 默认不在 Pull Request 中运行。
- 不输出响应原文。
- 只验证读取，不执行写操作。

### 28.6 安全测试

- 未登录访问所有 API
- 普通用户访问管理员 API
- CSRF
- Origin 绕过
- SSRF IPv4、IPv6、DNS Rebinding 和重定向
- SQL 注入
- JSONPath 资源消耗
- 日志敏感信息泄露
- Reveal 和导出权限
- 默认 Secret 缺失时拒绝启动

### 28.7 并发与恢复

- 单项失败不终止 Check All
- 定时刷新与手动刷新并发
- 刷新时修改配置
- 服务在事务中退出
- 服务重启恢复任务
- 同一连接不重复登录
- 告警通知失败后重试

## 二十九、验收标准

一个平台只有满足以下条件才标记 verified：

1. 能识别目标平台和版本。
2. 能使用文档要求的最小权限凭据认证。
3. 能正确读取账户余额或明确标记不支持。
4. 声称支持远端 Key 时，能完成分页同步。
5. 声称支持分组时，能读取分组并正确关联到 Key。
6. 账户余额不会因多个 Key 重复统计。
7. 无限、负余额和多币种状态正确。
8. 认证失败、429、5xx 和部分失败行为符合设计。
9. 不在日志、数据库非密文字段或前端列表泄露完整凭据。
10. 对应版本的契约 Fixture 和 E2E 测试通过。

核心模块验收：

- Check All 中一个供应商失败时其他供应商仍完成。
- 余额查询失败时保留最后一次成功数据并标记 stale。
- SQLite 事务不会被并发刷新和配置修改破坏。
- 未经管理员鉴权无法读取任何供应商数据。
- Custom 适配器无法访问未授权内网或云元数据地址。
- 自动化默认关闭，Dry Run 可以完整展示拟执行动作。

## 三十一、Docker Compose

~~~yaml
services:
  provider-monitor:
    profiles: [provider-monitor]
    build:
      context: ./provider-monitor
      args:
        NPM_REGISTRY: ${NPM_REGISTRY:-}
    container_name: sub2api-provider-monitor
    restart: unless-stopped
    ports:
      - "127.0.0.1:${PROVIDER_MONITOR_PORT:-9871}:3200"
    extra_hosts:
      - "host.docker.internal:host-gateway"
    environment:
      - PORT=3200
      - SUB2API_BASE_URL=${SUB2API_BASE_URL:-http://host.docker.internal:8080}
      - PROVIDER_MONITOR_SECRET=${PROVIDER_MONITOR_SECRET:?PROVIDER_MONITOR_SECRET is required}
      - PROVIDER_MONITOR_AUTH_MODE=${PROVIDER_MONITOR_AUTH_MODE:-sub2api}
      - PROVIDER_MONITOR_LOG_LEVEL=${PROVIDER_MONITOR_LOG_LEVEL:-info}
    volumes:
      - ./data/provider-monitor:/app/data
    healthcheck:
      test: ["CMD", "wget", "-q", "--spider", "http://localhost:3200/api/health"]
      interval: 30s
      timeout: 5s
      retries: 3
~~~

PROVIDER_MONITOR_SECRET 不提供默认值。首次部署必须由管理员生成随机 Secret。
